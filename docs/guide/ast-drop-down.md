# Authoring against the AST directly

When the `Keiki.Builder` DSL can't express what you need, drop down
to the underlying `Keiki.Core` AST and hand-author `Edge` records.
This guide is for that case.

For the canonical authoring path (which you should reach for first),
see the user-guide §3 and `Keiki.Builder`'s haddock. The two example
aggregates (`Jitsurei.UserRegistration`,
`Jitsurei.EmailDelivery`) ship the same transducer in both
forms — `userReg` / `userRegAST` and `emailDelivery` /
`emailDeliveryAST` — so you can read the AST form against a
familiar builder version line by line.

---

## 1. When to drop down

Three scenarios where the builder genuinely can't carry you:

1. **Bespoke guard logic.** A guard the builder's `requireEq` /
   `requireGuard` can't accumulate cleanly — typically a custom
   `HsPred` shape constructed by helper functions outside the
   builder, or guards that share a common sub-predicate you want
   to bind once and reuse.
2. **Helper-driven edge construction.** The aggregate's edges are
   built by parameterised helpers (e.g. `mkApprovalEdge`,
   `mkTimeoutEdge`) that produce `Edge` values directly. The
   builder is meant to *author* edges, not be a pluggable
   assembly tool.
3. **Side-by-side AST + builder for migration / parity tests.**
   The two example aggregates do this exactly: the AST form
   stays in the module as a reference for the builder's
   equivalence test (`*BuilderSpec`).

Most aggregates do not need this. Authoring in the AST loses the
distinct-target safety check (`Disjoint` on `combine`), the
finalize-time `goto` validation, and the field-keyed `emit`
record shape — three of the builder's load-bearing
ergonomics.

---

## 2. The shape of an AST aggregate

Both example aggregates follow the same module shape:

```haskell
yourAggregate :: SymTransducer (HsPred Regs Cmd) Regs Vertex Cmd Event
yourAggregate = SymTransducer
  { edgesOut    = yourAggregateEdges
  , initial     = StartingVertex
  , initialRegs = emptyRegs
  , isFinal     = \case TerminalVertex -> True; _ -> False
  }

yourAggregateEdges
  :: Vertex -> [Edge (HsPred Regs Cmd) Regs Cmd Event Vertex]
yourAggregateEdges = \case
  V1 -> [ Edge { … }, Edge { … } ]
  V2 -> [ Edge { … } ]
  V3 -> []
```

`SymTransducer` is a record. `edgesOut` is a vertex-keyed lookup
function the user supplies; the `\case` shape is the conventional
one. Vertices not listed default to `[]` (terminal).

Each `Edge` is a four-field record:

```haskell
Edge { guard  :: HsPred rs ci
     , update :: Update rs w ci   -- existential w; closes here
     , output :: Maybe (OutTerm rs ci co)
     , target :: s
     }
```

The `update` field's phantom `w :: [Symbol]` slot-set index closes
in the record (it's existentially scoped). That's the cost of
hand-authoring: you give up the static distinct-target check on
register writes.

## 3. The four fields, hand-authored

Reference: `Jitsurei.UserRegistration.userRegASTEdges` and
`Jitsurei.EmailDelivery.emailDeliveryASTEdges`.

### 3.1 `guard`

The smart constructors from `Keiki.Core`:

| Helper | Builds |
|---|---|
| `matchInCtor ic` | `PInCtor ic` — match on input constructor |
| `a .== b` | `PEq a b` |
| `PAnd p q`, `POr p q`, `PNot p` | the structural combinators |
| `PTop`, `PBot` | the units |

The standard guard shape combines `matchInCtor` with one or more
`PEq` predicates:

```haskell
guard = PAnd isConfirm
             (inpConfirm #confirmCode
               .== proj (#confirmCode :: Index UserRegRegs ConfirmationCode))
```

The `isConfirm` shorthand is generated by `deriveAggregateCtors`
(it's `matchInCtor inCtorConfirm` under the hood). Using it makes
the shape read as "this edge fires *on* `Confirm`, *and* the input
confirmCode equals the stored confirmCode."

The explicit `:: Index Regs T` annotation on `proj` is required —
the AST-level `proj` takes the existentially-typed `Index`, so the
slot's value type can't be inferred from `#name` alone.

### 3.2 `update`

The smart constructors from `Keiki.Core`:

| Helper | Effect |
|---|---|
| `UKeep` | identity update |
| `USet ix term` | write `term` into slot at `ix` |
| `combine u1 u2` (or backtick: `` u1 `combine` u2 ``) | sequence |

A multi-write update reads as a `combine` chain:

```haskell
update =
    USet (#email :: IndexN "email" Regs Email)
         (inpStart #email)
      `combine`
    USet (#confirmCode :: IndexN "confirmCode" Regs ConfirmationCode)
         (inpStart #confirmCode)
      `combine`
    USet (#registeredAt :: IndexN "registeredAt" Regs UTCTime)
         (inpStart #at)
```

The `IndexN` annotation on each slot pins its name and value type
— required because `combine`'s `Disjoint` constraint reads the
phantom `[Symbol]` index off each `USet`. Without the annotation,
the constraint can't see the slot name and the static check
silently no-ops.

`combine`'s `Disjoint` constraint *is* still active: writing
twice to the same slot fails to compile. You just have to
annotate every `USet`.

### 3.3 `output`

`Just (pack ic wc fs)` to emit, `Nothing` for an ε-edge.

```haskell
output = Just $ pack
            inCtorStart                            -- the InCtor
            wireRegistrationStarted                -- the WireCtor
            (OFCons (inpStart #email)              -- OutFields chain
              (OFCons (inpStart #confirmCode)
                (OFCons (inpStart #at) OFNil)))
```

The `OFCons … OFNil` chain is the heterogeneous list of
field-terms. The order matches the event payload's record-field
order (which is what `deriveWireCtors` walked at TH-derivation
time).

The `*:` / `oNil` operator form (re-exported from `Keiki.Core`) is
sugar for the same chain:

```haskell
inpStart #email *: inpStart #confirmCode *: inpStart #at *: oNil
```

The TH-derived `<Ctor>TermFields rs ci` record from `deriveWireCtors`
is **not** available in the bare AST form — it's part of the
`Keiki.Builder` `ToOutFields` typeclass machinery. If you want
field-keyed output construction, you need the builder.

### 3.4 `target`

The next vertex. Must be a value of the vertex enum:

```haskell
target = Confirmed
```

Unlike the builder, there's no `goto`-style validation — you set
the field directly and the type checker enforces it.

## 4. ε-edges in AST form

```haskell
Edge
  { guard  = isGdpr
  , update = USet (#deletedAt :: IndexN "deletedAt" Regs UTCTime)
                  (inpGdpr #at)
  , output = Nothing                  -- ε-edge
  , target = Deleted
  }
```

The only difference from a non-ε edge is `output = Nothing`.
`delta` advances the state, `omega` returns `Nothing`, and
through the `Decider` façade `decide` returns `[]` (see
user-guide §5.1).

## 5. What you keep, what you lose

### Kept

- **`solveOutput`** — mechanical inversion still works; the
  hand-authored `OPack` carries enough information to derive
  `applyEvent`.
- **`checkHiddenInputs`** — the build-time hidden-input check
  reads the `Edge` records directly, builder or not.
- **`isSingleValuedSym`** — single-valuedness is `BoolAlg`-
  polymorphic over the predicate carrier; lift with `withSymPred`
  the same way.
- **All evaluators** — `delta`, `omega`, `step`, `applyEvent`,
  `reconstitute` are agnostic to authoring path.
- **The `Decider`/`Acceptor` façades** — they consume
  `SymTransducer` directly.
- **`compose`** — operates on `SymTransducer` values regardless
  of how they were built.

### Lost

- **Static distinct-target check on register writes.** The
  `Disjoint` constraint on `combine` is still there, but writing
  `USet` without the `:: IndexN "name" Regs T` annotation hides
  the slot name from the constraint and the check silently
  passes a duplicate.
- **Finalize-time `goto` validation.** AST-form edges have a
  required `target` field — there's nothing to validate.
- **Field-keyed `emit` record.** `<Ctor>TermFields` is
  builder-only.
- **`PayloadProj` / `OverloadedRecordDot` for input fields.**
  `d.fieldName` requires the `PayloadProj` handle the builder
  threads. The AST form uses `inpFoo #fieldName` instead.

## 6. Mixing builder and AST in one module

Both forms produce `SymTransducer` values of the same type, so
you can keep them side by side. The two example aggregates
already do this — `userReg` is the builder form, `userRegAST` the
AST form. The builder spec
(`UserRegistrationBuilderSpec`) asserts the two are observationally
equivalent on a small fixture.

You can also mix at the *edge* level — author one transducer's
edges via builder helpers and concatenate with hand-authored
edges. Concretely: the builder produces an `Edge` value of the
same type the AST does, so a builder block that emits one edge
can be folded into an AST `\case` lookup. This is rare; the usual
move is to commit to one form per transducer.

## 7. A pre-flight checklist before dropping down

Before you reach for the AST, confirm:

- [ ] You've tried `requireGuard` for custom guard shapes. It
  takes any `HsPred rs ci` directly, including ones built by
  helper functions.
- [ ] You've tried passing helper-built `Edge` values to a thin
  builder wrapper that calls them. The builder's `from` / `onCmd`
  are not the only authoring path; you can construct `Edge`
  records and pass them through.
- [ ] You've checked the user-guide §3 and Builder haddock for an
  operation you missed.
- [ ] You're not just frustrated by the syntax. The
  `BlockArguments` + `QualifiedDo` + `OverloadedRecordDot` stack
  is unfamiliar; that's not a reason to drop the static checks
  the builder provides.

If you've cleared the checklist and still need the AST, write the
AST form. The two example aggregates' AST modules are 60–80 lines
each and read clearly.
