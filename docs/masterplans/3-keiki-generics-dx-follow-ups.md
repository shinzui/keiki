# Keiki.Generics DX follow-ups

This MasterPlan is a living document. The sections Progress, Surprises & Discoveries,
Decision Log, and Outcomes & Retrospective must be kept up to date as work proceeds.


## Vision & Scope

The keiki library shipped a v2 pure core under MasterPlan 2
(`docs/masterplans/2-retire-v1-escape-hatches-in-pure-core-tinpproj-sbv-boolalg.md`).
That work retired the v1 escape hatches (`TInpField` → `TInpCtorField`; the
hand-written `OPack` inverse → structural `solveOutput`; the best-effort
`isSingleValued` → SBV-backed symbolic decision) and introduced a small
ergonomics module `src/Keiki/Generics.hs` that walks `GHC.Generics`
representations to retire mechanical boilerplate at the example layer
(`mkInCtorVia`, `mkWireCtorVia`, `emptyRegFile`).

The follow-up design note `docs/research/keiki-generics-design.md`
catalogued **seven future improvements** in approximate-leverage order.
This MasterPlan delivers three of them — the three that share a single
theme (extending `Keiki.Generics` and the surrounding examples to push
authoring DX further while preserving existing guarantees). The other
four items are scheduled differently (see *Out of scope* below).

After all three child plans are complete, the repository contains:

- A Template-Haskell splice — `$(deriveAggregateCtors ''CmdSum
  ''RegFileSlots ...)` — that replaces the three top-level declarations
  per command constructor (`inCtor*`, `inp*`, `is*`) with one form,
  plus a `RegFieldsOf` type family that derives the slot-list type
  alias from a payload record's `Generic` representation. Together
  these retire ~50 additional lines per aggregate at the example layer.
- A `WitnessExtract` typeclass derived via `GHC.Generics` and a
  `symSatExt` operation in `Keiki.Symbolic` that materializes a concrete
  `(RegFile rs, ci)` witness from an SBV `sat` model. The User
  Registration aggregate gains a "build a concrete witness from a
  satisfiable predicate and verify `models` agrees" round-trip test,
  closing the gap EP-2 of MasterPlan 2 punted on (M5 placeholder
  `sat` returning `Just (RNil-coerced, opaque ci)`).
- A new module `src/Keiki/Decider.hs` exporting a `toDecider`
  projection from a `SymTransducer` to a Chassaing-shape
  `Decider c e s` record. `decide` is built on `omega`; `evolve` is
  built on `delta` + `applyEvent`. The agreement between forward and
  inverse — guaranteed by the keiki formalism — is asserted by a
  per-edge property test so users coming from naive-decider
  implementations can adopt the keiki core without writing both
  directions.
- An updated `docs/research/keiki-generics-design.md` whose Future
  Improvements section retires items A, B, D, and E (with C
  documented as the considered-and-rejected alternative); items F and
  G are explicitly punted with pointers (F to `docs/masterplans/4-...`,
  G to a future v3 MasterPlan).

The MasterPlan-level acceptance criterion: **the three test categories
above all pass (`cabal build && cabal test` green); the User
Registration aggregate's `inCtor*`/`inp*`/`is*` declarations are
generated by the TH splice; the symbolic `sat` round-trip test on
`userReg` returns `Just (regs, cmd)` with `models p (regs, cmd) == True`;
and a `Decider`-shape adapter for `userReg` produces the same event
sequence on the canonical log as `omega`.**

In scope:

- TH-driven aggregate-ctor scaffolding + `FieldsOf`/`RegFieldsOf`
  type family (item A and item B from `keiki-generics-design.md`).
- Generic-derived `WitnessExtract` for `symSatExt` (item D).
- A new `Keiki.Decider` façade module (item E).
- Migration of `Keiki.Examples.UserRegistration` (V5) to the TH
  surface and to use `symSatExt` and `toDecider` in tests; V0 stays
  on the existing surface (its hidden-input demonstration is
  orthogonal to the DX work).
- Updates to `docs/research/keiki-generics-design.md` after each EP
  to retire the corresponding "Future improvements" entry.

Out of scope:

- **Item C** (`HasInpHelpers` typeclass for `Term` projection
  helpers). The design note already states: "Probably not worth it
  without (A) — TH does the same job more directly." EP-8's design
  milestone records C as a considered-and-rejected alternative; no
  code is written for it.
- **Item F** (crem-style composition combinators on
  `SymTransducer`). The design note explicitly says F "doesn't
  affect the per-ctor DX" and is "independent of `Keiki.Generics`'
  DX scope." MasterPlan 4
  (`docs/masterplans/4-composition-combinators-on-symtransducer.md`)
  owns it. The MP-4 design milestone may further fan out into
  per-combinator EPs (`compose`, `feedback`, `alternative`).
- **Item G** (compile-time topology safety via a
  `SymTransducerStrict` parameterized over a type-level topology).
  The design note explicitly says this "is a separate
  MasterPlan-sized initiative" appropriate as a v3 direction. Defer
  to a future MasterPlan; record the deferral in the Decision Log.
- **Standalone EP-7** (GHC 9.12 upgrade,
  `docs/plans/7-upgrade-keiki-to-ghc-9-12.md`) is *not* a child of
  this MasterPlan, but it is a recommended-soft-dep for all three
  children (the TH and Generic dispatches benefit from boot-library
  bumps, and CI alignment with sibling projects like `mori`,
  `notion-hub`, and `hw-kafka-streamly` reduces friction). Each
  child EP's M0 milestone records the GHC version it was
  implemented under.
- The Order Fulfillment process manager (synthesis §5) — runtime
  smoke test, deferred to the future runtime MasterPlan.
- Changes to `Keiki.Symbolic`'s symbolic translation beyond what
  EP-9 needs for witness extraction (`PMatchC` translation,
  unbounded-`Text` strategies, etc. stay as they are).


## Decomposition Strategy

The initiative decomposes into **three child ExecPlans**, one per
in-scope future-improvement entry from the design note's "Future
improvements" list:

- **EP-8** delivers items A + B together: the TH splice plus the
  `RegFieldsOf` type family. They are bundled because B is a small
  mechanical helper that A's splice naturally consumes (the splice
  emits an `InCtor _ (RegFieldsOf PayloadRecord)` signature when B
  is available, an explicit `'[ '("name", T), ... ]` list otherwise).
  Splitting them would force a coordination dance about which form
  the splice emits; bundling makes B the splice's substrate.
- **EP-9** delivers item D: a `Generic`-driven `WitnessExtract`
  instance derivation that lets `Keiki.Symbolic.symSatExt`
  materialize a concrete `(RegFile rs, ci)` from an SBV `sat` model.
  This is the "M5 placeholder we punted on in EP-2 of MasterPlan 2"
  follow-up.
- **EP-10** delivers item E: a `Keiki.Decider` façade module
  exposing `toDecider :: SymTransducer ... -> Decider c e s`. A
  thin, additive piece of plumbing.

Sharpening is folded into each child as an early "design
exploration" milestone (M1) that produces a focused note, mirroring
MasterPlan 2's pattern. The notes update the existing
`docs/research/keiki-generics-design.md` rather than creating new
files (each EP's M1 amends the corresponding "Future improvements"
entry with the chosen shape and rationale).

**Why three plans, not one:**

Each item touches a different sub-system and has a different
validation gate:

- EP-8 lives at the TH/`Keiki.Generics` boundary. Acceptance is a
  green build with the User Registration aggregate's
  `inCtor*`/`inp*`/`is*` declarations replaced by a single splice.
- EP-9 lives in `Keiki.Symbolic` + `Keiki.Generics`. Acceptance is a
  symbolic `sat` round-trip on `userReg` returning a concrete
  witness that `models` accepts.
- EP-10 lives in a new module `Keiki.Decider`. Acceptance is a
  property test asserting `toDecider userReg`'s `decide`/`evolve`
  reproduce the canonical log.

A single ExecPlan covering all three would conflate three distinct
gates and force one to land without proving anything.

**Why three plans, not four (item C):**

Item C (`HasInpHelpers` typeclass) is documented in
`docs/research/keiki-generics-design.md` as "Probably not worth it
without (A) — TH does the same job more directly." Separating C as
its own EP means a separate validation gate for a code path that A
supersedes. EP-8's M1 design milestone records C as a
considered-and-rejected alternative and explains why; no code is
written for it. This decision is recorded in the Decision Log below.

**Alternatives considered:**

- *Single ExecPlan covering all three.* Rejected for the reasons
  above: distinct subsystems, distinct validation gates.
- *Add an EP-Cancel item for C, building the typeclass alongside A
  for back-compat.* Rejected: A's splice generates the same three
  declarations C's typeclass would expose at use sites (`inp
  @"StartRegistration" #email` vs `inpStart #email`), so C provides
  no leverage over A. The design note's own analysis suffices.
- *Bundle in items F (combinators) or G (topology safety).*
  Rejected: F is its own MP-4; G is a v3 MasterPlan-sized initiative
  per the design note. Bundling them here would balloon scope and
  conflate themes.
- *Bundle in EP-7 (GHC 9.12 upgrade) as a child.* Rejected: EP-7 is
  a tooling concern with zero shared coordination surface with the
  DX work. It runs as a standalone ExecPlan.

**Phasing:**

- *Phase 1 (optional, recommended).* EP-7 lands first to align
  tooling with sibling projects.
- *Phase 2 — three children, parallel.* EP-8, EP-9, EP-10 proceed
  in any order. Sequential by a single contributor is the expected
  mode; parallel by multiple contributors is supported.

A single contributor implementing all three plans in sequence is
the expected mode. The order is flexible; a natural sequence is
EP-10 → EP-8 → EP-9 (smallest to largest), but any order works.


## Integration Points

The three child plans share four artifacts. This section names them
explicitly so decisions made in any one plan don't violate the
others' assumptions.

### IP-1: `keiki.cabal` build-depends and exposed-modules

**Plans involved:** EP-8 (adds `template-haskell` dep, possibly a
new `Keiki.Generics.TH` exposed module), EP-9 (no new dep but may
add a `Keiki.Generics.Witness` sub-module), EP-10 (adds a new
exposed module `Keiki.Decider` and a `keiki-test` other-module
`Keiki.DeciderSpec`).

**Owner:** each plan owns its cabal edit; the edits are additive
and don't conflict at the line level.

**Coordination rule:** if two children land back-to-back, the
second contributor merges the first's cabal change before adding
their own. The cabal sections involved are:

    library
        exposed-modules:    Keiki.Core
                            Keiki.Generics
                            Keiki.Symbolic
                            ... (additions per EP)
        build-depends:      base ^>= 4.20,
                            sbv >= 11.7 && < 15,
                            text ^>= 2.1,
                            time ^>= 1.12,
                            ... (template-haskell from EP-8)

EP-7 (external) may bump `base` to `^>= 4.21` (GHC 9.12 ships
base 4.21). The children's cabal edits are independent of that.

### IP-2: `Keiki.Examples.UserRegistration` migration target

**Plans involved:** EP-8 migrates the per-constructor declarations
to the TH splice (replacing `inCtorStart`, `inpStart`, `isStart`,
... × 5 with one splice form). EP-9 adds a symbolic round-trip
test that uses `userReg`'s `withSymPred` form. EP-10 adds a
property test using `toDecider userReg` against the canonical log.

**Owner:** EP-8 owns the source migration. EP-9 and EP-10 own
their respective test additions.

**Coordination rule:** EP-8 is the only EP that *modifies*
`UserRegistration.hs`; EP-9 and EP-10 *add tests* that import the
module's existing exports (`userReg`, `inCtorStart`, etc.). If EP-8
lands first, the exports are still available (the splice generates
the same names). If EP-9 or EP-10 lands first, no changes to
`UserRegistration.hs` are required — they just import what's already
exported. EP-8's M5 migration milestone explicitly verifies that
the post-splice export list still matches the pre-splice one
(name-by-name).

### IP-3: `docs/research/keiki-generics-design.md`

**Plans involved:** all three children edit the "Future improvements"
section. Each EP's M1 milestone amends the corresponding entry with
the chosen shape; each EP's terminal milestone marks the entry as
**Implemented (see EP-N)**.

**Owner:** each plan owns the entry it implements.

**Coordination rule:** the entries are clearly labelled (A, B, D, E)
and the edits target distinct subsections. Merge conflicts on this
file are mechanical and easy to resolve.

### IP-4: `Keiki.Generics` module surface

**Plans involved:** EP-8 may add a TH sub-module
(`Keiki.Generics.TH`) or extend the existing `Keiki.Generics`
module. EP-9 may add a witness-extraction sub-module
(`Keiki.Generics.Witness`) or extend the existing module.

**Owner:** EP-8 picks first (its M1 design milestone decides
sub-module-vs-flat). EP-9 follows whatever shape EP-8 picks if EP-8
has landed; otherwise EP-9 picks its own and EP-8 reconciles when it
lands (EP-8's M5 migration includes a "if EP-9 has split the module,
preserve its split" check).

**Coordination rule:** the default — when no constraints apply — is
"flat module, additive exports." Sub-module splits only happen if
the EP's M1 milestone documents a clear reason (large surface,
unrelated dep tree, expected reuse).

## Surprises & Discoveries

Document cross-plan insights, dependency changes, scope adjustments,
or unexpected interactions between child plans. Provide concise
evidence.

- **EP-10 forced a one-line Core export change** (2026-05-01). The
  EP-10 plan listed `Keiki.Core.applyEvent` as a consumed
  function, but the symbol was not in `Keiki.Core`'s export list
  (it was haddocked as "Internal helper for `reconstitute`"). EP-10
  added it to the `Pure-layer entry points` export group and
  expanded the haddock. Cross-plan implication for **EP-9** —
  whose `WitnessExtract` work also lives in `Keiki.Core`/`Keiki.Symbolic`
  — is mild: EP-9 will land on top of an `applyEvent`-exporting
  `Keiki.Core` and need not relitigate the export.

- **z3 is missing from the dev shell** (2026-05-01). `flake.nix`'s
  `devShells.default` does not include `pkgs.z3`, so `cabal test
  all` outside `nix-shell -p z3` produces 25 SBV-backed failures.
  Cross-plan implication: **EP-9**'s symbolic round-trip work
  also depends on z3. Adding z3 to the dev shell is an orthogonal
  ergonomic fix; EP-10 did not address it but flagged it for a
  possible follow-up.

- **Canonical command sequence is not a fixture** (2026-05-01).
  `Keiki.Examples.UserRegistrationSpec` records only the canonical
  *event log*, not the producing command sequence (because
  `reconstitute` works from events to state). EP-10 derived the
  matching command list locally for `test/Keiki/DeciderSpec.hs`.
  Cross-plan implication: **EP-9** will likely want the same
  fixture for symbolic-witness round-trips; a small refactor to
  expose a shared `canonicalCmds` from a test-helper module could
  remove the duplication when EP-9 lands. Not addressed by EP-10
  to stay in scope.

- **EP-8 line-count reduction is moderate (~16%), not the plan's
  ~30%** (2026-05-01). UserRegistration dropped from 403 → 339
  lines after migration. The plan estimated ~120 lines because it
  counted decl-lines uniformly; the actual pre-migration code was
  already condensed via `mkInCtorVia` (one line each), and the
  retained per-section haddock + headers absorbed the savings.
  The per-aggregate authoring win is real but a `wc -l` count
  understates it — for new aggregates the win is closer to "two
  splice forms vs. ~30 hand-written declarations". Cross-plan
  implication: none for EP-9; documented for retrospective
  accuracy.

- **`RegFieldsOf` lives in `Keiki.Generics`, not a sub-module**
  (2026-05-01). Per IP-4 of MP-3, the default is "flat module,
  additive exports". EP-8 followed it: `RegFieldsOf` + the
  `Keiki.Generics.TH` sub-module are the surface change. EP-9 may
  follow the same pattern (flat extension to `Keiki.Generics` for
  the `WitnessExtract` typeclass, no sub-module split) unless
  its M1 design milestone documents a clear reason to split.

  Rationale: The upgrade has zero shared coordination surface with

  Date: 2026-05-01

  Date: 2026-05-01
